# Промокоды в админ-панели

## Варианты реализации

### Вариант A: Supabase напрямую с фронта (реализован)

**Идея:** Вкладка «Промокоды» в админке обращается к Supabase из браузера. RLS уже разрешает `authenticated` читать и писать в `promo_codes`.

**Плюсы:**
- Не нужен бэкенд для админки.
- Один пароль админки (VITE_ADMIN_PASSWORD) + вход в Supabase (любой аккаунт с правами).

**Минусы:**
- Админ должен быть залогинен в Supabase (страница Account или форма входа), иначе запросы к `promo_codes` не пройдут по RLS.
- Любой залогиненный пользователь технически может писать в `promo_codes`; ограничение только паролем админки и тем, что путь /admin не рекламируется.

**Как работает:**
1. Админ заходит на /admin, вводит пароль (VITE_ADMIN_PASSWORD) → открываются вкладки «Отзывы» и «Промокоды».
2. На вкладке «Промокоды»: если нет сессии Supabase → показываем «Войдите в аккаунт» и ссылку на /account. Если сессия есть → загружаем список промокодов (`supabase.from('promo_codes').select()`), показываем форму создания и таблицу с действиями (удалить).
3. Создание: форма (код, тип скидки: % или фикс, значение, срок действия, макс. использований, описание) → `supabase.from('promo_codes').insert(...)`.
4. Удаление: кнопка → confirm → `supabase.from('promo_codes').delete().eq('id', id)`.

**Ограничения БД:** В таблице задаётся либо `discount_percent` (0–100), либо `discount_fixed` (число), не оба сразу.

---

### Вариант B: Бэкенд API (альтернатива)

**Идея:** На pb_backend эндпоинты вида `GET/POST/PATCH/DELETE /v1/promo`, проверка заголовка `X-Admin-Password`. Бэкенд с `SUPABASE_SERVICE_ROLE_KEY` делает CRUD в Supabase.

**Плюсы:** Service role не в браузере; админу не нужен аккаунт Supabase.  
**Минусы:** Нужен поднятый бэкенд и env (ADMIN_PASSWORD, SUPABASE_URL, SUPABASE_SERVICE_ROLE_KEY).

Для демо выбран вариант A; при необходимости позже можно добавить вариант B.

## Структура UI (вкладка «Промокоды»)

1. **Проверка сессии**  
   Если нет `supabase.auth.getSession()` → блок: «Войдите в аккаунт для управления промокодами» + ссылка на /account.

2. **Форма создания**
   - Код (обязательно, текст).
   - Тип скидки: «Процент» / «Фиксированная сумма».
   - Значение: число (для % — 0–100, для фикс — ≥ 0).
   - Действует до (дата, необязательно).
   - Макс. использований (число, необязательно).
   - Описание (необязательно).
   Кнопка «Создать» → валидация → insert в Supabase → обновить список.

3. **Таблица промокодов**
   Колонки: Код, Скидка, Действует до, Использовано / макс., Создан, Действия (Удалить).  
   Удаление с подтверждением.

4. **Редактирование**  
   Для MVP можно не делать; при необходимости — модалка с той же формой и PATCH по `id`.

---

## Переменные окружения

- **Фронт:** `VITE_ADMIN_PASSWORD` — пароль входа в админку (уже есть).  
- **Supabase:** `VITE_SUPABASE_URL`, `VITE_SUPABASE_ANON_KEY` — для запросов к `promo_codes` от имени залогиненного пользователя.
